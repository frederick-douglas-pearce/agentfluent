# PRD: Tool name normalization for SDK version boundaries

**Status:** Draft
**Date:** 2026-05-21
**Author:** PM Agent
**Research origin:** C-008 (anthropic-feature-watch pipeline)
**Decision log:** See `decisions.md` for referenced decisions.

---

## 1. Problem Statement

The TypeScript Agent SDK v0.3.142 (May 2026) removed the deprecated `TodoWrite` tool and replaced it with four Task tools: `TaskCreate`, `TaskUpdate`, `TaskGet`, `TaskList`. Sessions generated by agents running v0.3.142+ use the new tool names in `tool_use` content blocks. This creates three concrete problems for AgentFluent:

1. **Tool frequency fragmentation.** `analytics/tools.py:compute_tool_metrics()` counts tool names as raw strings. A user comparing pre- and post-upgrade sessions sees `TodoWrite` and `TaskCreate`/`TaskUpdate`/`TaskGet`/`TaskList` as unrelated tools rather than as the same capability under a new name.

2. **Diff false positives.** `agentfluent diff` comparing a baseline snapshot (pre-v0.3.142) against a current run (post-v0.3.142) will show `TodoWrite` disappearing and four new tools appearing. This looks like a behavioral regression but is actually a version-driven rename. There is no annotation mechanism to distinguish known renames from genuine tool-usage changes.

3. **Category gap.** `WRITE_TOOLS` in `agents/models.py` classifies state-changing tools for complexity assessment. The Task tools are not state-changing in the same way (they manage async task queues, not files), but they also lack any category. No "task-management" classification exists, so diagnostics modules that reason about tool roles (delegation complexity, workload assessment) have no way to recognize these tools.

### What already exists

- **Glossary:** `terms.yaml` line 840-844 has a `Task` entry with `aliases: [TaskCreate, TaskUpdate, TaskList, TaskGet, TaskOutput, TaskStop]`. This is the glossary's explanatory view but is not consumed by any analytics or diagnostics code path.
- **`WRITE_TOOLS` frozenset:** The pattern for tool-category constants exists. Adding a parallel `TASK_TOOLS` frozenset is a one-line addition.
- **No `TodoWrite` references:** The codebase has zero references to `TodoWrite`. This is purely a forward-looking normalization -- no existing logic needs migration.

## 2. Scope

### In Scope

1. **Known-rename registry** -- a module-level constant dict mapping deprecated tool names to their current equivalents. Seeded with `{"TodoWrite": "TaskCreate"}` (the primary mapping; `TodoWrite` was a single tool that created tasks, closest to `TaskCreate`).

2. **Tool name normalization in analytics** -- `compute_tool_metrics()` applies the rename map before counting, so pre- and post-rename sessions produce comparable frequency tables.

3. **`TASK_TOOLS` frozenset** -- parallel to `WRITE_TOOLS`, classifying `TaskCreate`, `TaskUpdate`, `TaskGet`, `TaskList`, `TaskOutput`, `TaskStop` as task-management tools. Available to diagnostics modules.

4. **Diff known-rename annotation** -- when `agentfluent diff` detects that a tool present in the baseline but absent in the current (or vice versa) is in the known-rename registry, annotate the diff output rather than silently treating it as a new/disappeared tool.

5. **Glossary update** -- add `TodoWrite` as a deprecated alias on the existing `Task` glossary entry with a note about the SDK version boundary.

### Out of Scope

- **Generic tool taxonomy framework.** No category hierarchy, no category enum, no category-aware grouping in the `analyze` output. A simple frozenset constant is sufficient. If future candidates require richer categorization, that work builds on top of the frozenset without breaking it.
- **Tool frequency grouping by category** in CLI output (e.g., collapsing all Task tools into one row). This is a display concern for a future story.
- **Config scanner tool-name validation.** Checking whether an agent's `tools:` frontmatter references deprecated names (`TodoWrite`) is a config-assessment feature, not an analytics feature. Deferred.
- **MCP tool rename tracking.** MCP tools use `mcp__<server>__<tool>` namespacing; renames there are server-specific. Different problem, different solution.
- **Normalizing `TaskOutput`/`TaskStop`** into the known-rename registry. These are new tools with no deprecated predecessor; they just need to be in `TASK_TOOLS`.

## 3. User Stories

### Story 1: Known-rename registry + analytics normalization

**As** a developer comparing agent sessions across an SDK upgrade boundary,
**I want** tool frequency counts to normalize deprecated tool names to their current equivalents,
**So that** I see `TaskCreate: 5` instead of `TodoWrite: 3, TaskCreate: 2` when the same agent ran before and after the upgrade.

### Story 2: `TASK_TOOLS` frozenset

**As** an AgentFluent diagnostics developer,
**I want** a recognized `TASK_TOOLS` constant parallel to `WRITE_TOOLS`,
**So that** diagnostics modules can classify task-management tool usage without hardcoding tool names inline.

### Story 3: Diff known-rename annotation

**As** a developer using `agentfluent diff` to compare a pre-upgrade baseline against a post-upgrade current run,
**I want** the diff to annotate tool-name changes from known renames rather than reporting them as new/disappeared tools,
**So that** I can distinguish version-driven tool renames from genuine behavioral changes.

### Story 4: Glossary deprecation note

**As** a user running `agentfluent explain Task`,
**I want** the glossary to mention that `TodoWrite` was the deprecated predecessor,
**So that** I understand the lineage when I encounter `TodoWrite` in older session data.

## 4. Design

### 4.1 Known-rename registry

Location: `src/agentfluent/agents/models.py` (alongside `WRITE_TOOLS` and `BUILTIN_AGENT_TYPES`).

```python
# Known tool renames across SDK versions. Keys are deprecated names;
# values are the current canonical name. Used by analytics normalization
# and diff annotation. Update this dict as Anthropic renames tools.
TOOL_RENAMES: dict[str, str] = {
    "TodoWrite": "TaskCreate",
}
```

Design rationale: a simple dict is the minimal data structure that supports both normalization (look up old name, get new name) and diff annotation (check whether a disappeared/appeared tool is a known rename). No classes, no registry pattern, no versioned history. The dict grows by one line per rename.

### 4.2 `TASK_TOOLS` frozenset

Location: `src/agentfluent/agents/models.py`.

```python
TASK_TOOLS: frozenset[str] = frozenset(
    {"TaskCreate", "TaskUpdate", "TaskGet", "TaskList", "TaskOutput", "TaskStop"},
)
```

Follows the `WRITE_TOOLS` pattern exactly. Not consumed by any diagnostics rule in this epic -- this is infrastructure for future stories (e.g., tool-inventory diagnostics #371, delegation complexity refinement).

### 4.3 Analytics normalization

In `analytics/tools.py:compute_tool_metrics()`, apply `TOOL_RENAMES` before aggregation:

```python
from agentfluent.agents.models import TOOL_RENAMES

# Inside the counting loop:
raw_name = block.name
canonical_name = TOOL_RENAMES.get(raw_name, raw_name)
counts[canonical_name] = counts.get(canonical_name, 0) + 1
```

This normalizes at the analytics layer, not the parser layer. The parser preserves raw tool names (they are data); the analytics layer normalizes for comparison. Downstream consumers of `ToolMetrics.tool_frequency` see canonical names.

### 4.4 Diff known-rename annotation

The diff tool-comparison currently happens implicitly via `_diff_agent_metrics` (per-agent-type deltas) but not via per-tool-name deltas. Tool frequency is not a first-class diff surface today.

**Minimal approach for this epic:** Since tool frequency is not yet diffed per-tool, the known-rename annotation applies where tool names surface in the diff:

1. **Recommendation deltas:** If a recommendation references a tool name (e.g., `tool_name` in `signal.detail`), and that tool name is in `TOOL_RENAMES`, the diff annotation notes the rename. This is handled by adding a `known_renames` field to `DiffResult` that lists any renames detected between the baseline and current envelopes.

2. **JSON output:** New optional field `known_tool_renames: list[{old: str, new: str}]` on `DiffResult`, populated when the baseline envelope contains tool names that map to different names in the current envelope via `TOOL_RENAMES`.

3. **CLI output:** A single info line in the diff summary when known renames are detected: `"[info] Known tool rename detected: TodoWrite -> TaskCreate (SDK v0.3.142+). Tool metrics are normalized."`.

### 4.5 Glossary update

Add `TodoWrite` to the `Task` entry's aliases list and add a deprecation note to the `long` field:

```yaml
- name: Task
  category: builtin_tool
  short: |
    Async task management (TaskCreate, TaskUpdate, TaskList, TaskGet, etc.).
  aliases: [TaskCreate, TaskUpdate, TaskList, TaskGet, TaskOutput, TaskStop, TodoWrite]
  long: |
    `TodoWrite` was the original single tool for task management, deprecated
    and removed in TypeScript Agent SDK v0.3.142 (May 2026). It was replaced
    by the granular Task tools: `TaskCreate`, `TaskUpdate`, `TaskGet`,
    `TaskList`, `TaskOutput`, `TaskStop`. AgentFluent normalizes `TodoWrite`
    to `TaskCreate` in analytics output for cross-version comparability.
```

## 8. Risks and Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| `TodoWrite -> TaskCreate` is an imperfect mapping (TodoWrite may have covered update semantics too) | Low -- affects legacy sessions only; worst case is slightly overcounting TaskCreate | Document in the `TOOL_RENAMES` comment. If a user reports inaccuracy, add a `TodoWrite -> Task` group mapping in a follow-up. |
| Future tool renames break analytics continuity | Low | `TOOL_RENAMES` dict is the single place to add new entries. The pattern established here makes future additions trivial. |
| `DiffResult` schema addition is a minor breaking change for JSON consumers | Low | `known_tool_renames` defaults to empty list. Existing consumers that don't read it are unaffected. |

## 9. Future Work (Explicitly Deferred)

- **Config scanner: deprecated tool-name detection.** Check agent `tools:` frontmatter for `TodoWrite` and recommend updating to `TaskCreate`. Would be a new `ConfigRecommendation` in the config scorer.
- **Tool frequency grouping by category.** Collapse `TaskCreate`/`TaskUpdate`/`TaskGet`/`TaskList` into a single "Task management" row in `analyze` output. Display concern, not analytics concern.
- **Per-tool diff table.** Full tool-by-tool frequency comparison in `agentfluent diff`. Currently only recommendations and token metrics are diffed per-item. Tool frequency diff would consume the normalization from S1 and the annotation from S3.
