# AgentFluent Glossary

A reference for the vocabulary that appears in `agentfluent` CLI output and JSON.
Every term used in `analyze --diagnostics` and `config-check` output is defined
here. Threshold values cited below match the live source as of this file's
last update ‚Äî Phase 2 (#191) will load them programmatically so the docs can't drift.

## Reading guide

`agentfluent` reports on three layers of agent behavior, and the glossary is
organized to match them:

1. **Execution analytics** ‚Äî what happened: token counts, tool calls, costs.
   Mostly familiar Anthropic API vocabulary, plus a few AgentFluent-specific
   rollups (e.g., `cache_efficiency`). Skim the **Token types** section if
   you've worked with the Anthropic API before; otherwise read it through.

2. **Behavior diagnostics** ‚Äî why something looks off: signal types, severity,
   confidence. This is AgentFluent-invented vocabulary with no analogue in
   external Claude Code documentation. **Start here if you're a first-time
   user** ‚Äî the **Signal types**, **Severity**, and **Confidence tier**
   sections together give you the mental model needed to act on `--diagnostics`
   output.

3. **Recommendations** ‚Äî what to change: targets, agent concerns, model-routing
   tiers. These map signals to specific configuration surfaces. The
   **Recommendation target** and **Built-in agent concern** sections explain
   why two findings on the same agent might suggest different fixes.

If you're triaging a single piece of output, the lookup-friendly section
headings work as ctrl-F anchors. Phase 2 (#191) will add `agentfluent explain
<term>` for terminal-native lookup without opening this file.

## Token types

The Anthropic API splits each request's tokens into four categories with
different per-token costs. AgentFluent reports each separately so you can
diagnose where spend concentrates.

### `input`

**Short:** Tokens read from the prompt that aren't served from a cache.

**Detail:** Charged at the model's standard input rate. Every API call accrues
some `input` tokens ‚Äî the system prompt, the latest user turn, and any newly
appended context that didn't exist in the prior cached prefix. Sustained
high `input` ratios (vs. `cache_read`) on long sessions usually indicate the
prompt prefix is changing in ways that defeat caching.

**Example:** `agentfluent analyze` reports `Input tokens: 12,840` per session.

**Related:** [`cache_read`](#cache_read) (the cached complement), `cache_efficiency`.

### `output`

**Short:** Tokens generated by the model.

**Detail:** Charged at the model's standard output rate (typically 5√ó the
input rate). Output is the only token category Claude bills for *generating*;
all others bill for *reading* tokens that were already in the prompt. High
output ratios usually mean the agent is producing long-form content (code
generation, prose, structured tool calls); low ratios with high spend mean
prompt-side cost dominates.

**Example:** `Output tokens: 8,420` in the analyze table.

**Related:** `total_cost` (where [`output`](#output) typically dominates per-call cost).

### `cache_creation`

**Short:** Tokens written to the prompt cache for later reuse.

**Detail:** Charged at 1.25√ó the input rate (the 5-minute TTL tier). Pays
the upfront cost so subsequent calls within the cache window can read the
same prefix at the heavily-discounted `cache_read` rate (~10% of input).
First call in a session typically writes; later calls read. A `cache_creation`
spike on a session that should be re-using a stable prefix indicates cache
invalidation ‚Äî usually because the system prompt or early turns are changing.

**What's typically in the cached prefix:** the system prompt scaffold,
tool definitions, and `CLAUDE.md` content all land in the cached side
on a typical Claude Code session. The first turn writes them; every
subsequent turn within the cache window reads them. Anything that
*changes* on every turn (the conversation history) is on the
non-cached side.

**Cache-invalidation triggers** that push these tokens back to
`cache_creation`:
- TTL expires (default 5-minute gap between turns)
- `CLAUDE.md` is edited mid-session
- Any earlier prefix bytes change (rare for `CLAUDE.md`, which sits
  near the front)

**Example:** `Cache creation tokens: 14,450` on the first call of a long
session; `0` on subsequent calls until the cache TTL expires or the
prefix changes.

**Related:** [`cache_read`](#cache_read), `cache_efficiency`.

### `cache_read`

**Short:** Tokens read from the prompt cache.

**Detail:** Charged at 10% of the input rate (the heavily-discounted tier).
This is the dominant token category by volume on long sessions and the
reason cached agents are 10‚Äì20√ó cheaper than uncached. The `cache_efficiency`
metric (`cache_read / total_input`) tells you what fraction of every prompt
is hitting the cache; ratios below ~80% on long sessions warrant
investigation.

**What you're reading from cache:** the system prompt scaffold, tool
definitions, and `CLAUDE.md` content (whatever was written by the prior
turn's `cache_creation`). Each subsequent turn reads the same prefix
bytes from cache instead of re-paying the full input rate.

**Why `cache_efficiency` drops on a long session:** the cache TTL
(default 5 minutes) expired between turns, or `CLAUDE.md` was edited,
or some earlier prefix byte changed. Each invalidation forces the next
turn to write `cache_creation` tokens before any `cache_read` resumes.

**Example:** `Cache read tokens: 1,234,567` ‚Äî the hot path on a session
that's been running for an hour.

**Related:** [`cache_creation`](#cache_creation), `cache_efficiency`.

## Signal types

A **signal** is an observed behavior pattern that AgentFluent flagged as
worth attention. Signals are *not* errors ‚Äî they're hypotheses about agent
behavior that the correlator turns into actionable recommendations. Each
signal has a fixed severity and detection threshold. Custom (user-defined)
agents and built-in (Claude Code) agents both emit signals, but the
recommendations differ ‚Äî see [**Built-in agent concern**](#built-in-agent-concern).

### `token_outlier`

**Short:** An invocation consumes far more tokens per tool call than the
agent's typical run.

**Detail:** Triggered when `tokens_per_tool_use` exceeds `OUTLIER_THRESHOLD`
√ó the per-agent-type mean (currently **2.0√ó**). Computed across all
invocations of the same agent type within the analyzed sessions, so two
invocations of `pm` are needed before any can be flagged. Indicates the
agent is over-fetching context, exploring unfocused, or being given an
ambiguous prompt.

**Example:**
```
Agent 'Explore' has 40,288 tokens/tool_use, 8.0x above the 5,064 mean.
```

**Severity:** `warning`.

**Related:** [`duration_outlier`](#duration_outlier), [`stuck_pattern`](#stuck_pattern) (often co-occur),
[`recommendation target: prompt`](#prompt).

### `duration_outlier`

**Short:** An invocation takes far longer per tool call than the agent's
typical run.

**Detail:** Same threshold as `token_outlier` (`OUTLIER_THRESHOLD = 2.0√ó`
the per-agent mean) but on `duration_per_tool_use`. Often co-occurs with
`token_outlier`. When it appears alone, the agent is making slow tool
calls (network latency, large file reads) rather than over-thinking.

**Example:**
```
Agent 'pm' has 45.2s/tool_use, 3.1x above the 14.6s mean.
```

**Severity:** `warning`.

**Related:** [`token_outlier`](#token_outlier), [`model_mismatch`](#model_mismatch) (an underspec'd model often
shows here).

### `tool_error_sequence`

**Short:** Consecutive tool calls inside a subagent trace returned errors.

**Detail:** Triggered by 2+ consecutive `is_error: true` results from the
same subagent. Escalates to `critical` at 4+. Excludes errors already
covered by `retry_loop` or `stuck_pattern` so the same evidence doesn't
fire two signals. Indicates the agent is missing fallback instructions ‚Äî
it ran into a wall and kept walking into it.

**Example:**
```
Subagent 'general-purpose' had 3 consecutive tool errors.
```

**Severity:** `warning` (2‚Äì3 errors); `critical` (4+).

**Related:** [`retry_loop`](#retry_loop), [`stuck_pattern`](#stuck_pattern), [`permission_failure`](#permission_failure).

### `retry_loop`

**Short:** A subagent retried the same tool 3+ times in a row.

**Detail:** Triggered by a `RetrySequence` with `attempts ‚â• 3` that doesn't
qualify as `stuck_pattern`. Indicates missing error-recovery guidance ‚Äî the
agent is correctly identifying failure but doesn't know what to try
differently. The retries may eventually succeed, in which case
`eventual_success: true` is recorded in the signal detail.

**Example:**
```
Subagent 'architect' retried tool 'Read' 5 times.
```

**Severity:** `warning`.

**Related:** [`stuck_pattern`](#stuck_pattern) (the more-severe variant), [`tool_error_sequence`](#tool_error_sequence).

### `stuck_pattern`

**Short:** A subagent retried the same tool with **identical input** 4+
times ‚Äî the most severe retry signal.

**Detail:** Triggered when 4+ retry attempts share byte-identical
`input_summary`. The agent isn't varying its approach between attempts;
it's stuck in a loop. Emits `critical` because nothing the agent can do
will recover without prompt-level intervention.

**Example:**
```
Subagent 'Explore' repeated tool 'Bash' with identical input 6 times
without progress.
```

**Severity:** `critical`.

**Related:** [`retry_loop`](#retry_loop) (the less-severe variant where input varied).

### `permission_failure`

**Short:** A tool result contained a permission-denied keyword.

**Detail:** Triggered when a subagent tool result matches one of:
`permission denied`, `access denied`, `not allowed`, `blocked`. Emitted
once per unique tool denied (not per call), so the recommendation maps
to a single config change ("grant tool X"). Cross-cuts with
`tool_error_sequence` ‚Äî both fire from the same evidence ‚Äî because the
remediation surfaces are different (tool config vs. prompt fallback).

**Example:**
```
Subagent 'pm' was denied access to tool 'Write' ('permission denied').
```

**Severity:** `critical`.

**Related:** [`error_pattern`](#error_pattern), [`recommendation target: tools`](#tools).

### `error_pattern`

**Short:** Free-text error keywords detected in an invocation's final output.

**Detail:** Lower-fidelity than the trace-level signals above. Scans
`output_text` for `blocked`, `permission denied`, `failed`, `error`,
`retry`, `unable to`, `not found`, `timed out`. Suppressed for any agent
type that already has a trace-level signal ‚Äî trace evidence is strictly
more informative than a keyword scan.

**Example:**
```
Agent 'pm' output contains 'unable to'.
```

**Severity:** `critical` for permission-denied keywords, `warning` otherwise.

**Related:** [`permission_failure`](#permission_failure), [`tool_error_sequence`](#tool_error_sequence) (both supersede
`error_pattern` when present for the same agent).

### `model_mismatch`

**Short:** An agent's declared model is wrong for the complexity of its
observed work.

**Detail:** Per-agent-type signal. Compares the agent's declared model
family (`current_model`) against a complexity classification computed from
its mean tool calls, mean tokens, error rate, and write-tool usage.
Emits `overspec` when a complex model is doing simple work (potential
savings) or `underspec` when a simple model is doing complex work
(potential failures). Only fires when the agent has 3+ invocations and
declares a model in its `.claude/agents/X.md` frontmatter.

**Example:**
```
Agent 'pm' is overspec'd: classified 'simple' but uses claude-opus-4-7.
Estimated savings $0.42 across 11 invocations by switching to
claude-haiku-4-5.
```

**Severity:** `warning`.

**Related:** [`complexity_tier`](#complexity_tier), [`overspec`](#overspec), [`underspec`](#underspec),
[`recommendation target: model`](#model).

### `mcp_unused_server`

**Short:** An MCP server is configured but never called in observed sessions.

**Detail:** Triggered when an MCP server appears in `~/.claude.json`
or `.mcp.json` but no tool calls reference it across the analyzed
sessions. May be configured for projects you didn't analyze; consider
the `info` severity. Cross-cutting signal ‚Äî `agent_type` is `null` since
MCP servers aren't scoped to one agent.

**Example:**
```
MCP server 'slack' is configured in /home/u/.claude.json but has 0 tool
calls across 10 analyzed sessions. Consider removing from mcpServers or
marking as disabled.
```

**Severity:** `info`.

**Related:** [`mcp_missing_server`](#mcp_missing_server).

### `mcp_missing_server`

**Short:** An MCP server is called but not configured.

**Detail:** Triggered when observed tool calls reference an MCP server
that isn't in any discovered MCP configuration. Often the result of a
project-scoped server in a project you analyzed but is not the active
project. Cross-cutting (`agent_type: null`).

**Example:**
```
Observed 5 calls to MCP server 'github' but it is not in any discovered
MCP config. Add to ~/.claude.json or .mcp.json.
```

**Severity:** `warning`.

**Related:** [`mcp_unused_server`](#mcp_unused_server).

## Severity

A flat 3-level scale used by signals and recommendations. Severity affects
the table render order (critical first) and the row color in CLI output.

### `critical`

**Short:** A finding that almost certainly indicates broken behavior ‚Äî the
agent failed or is stuck.

**Detail:** Reserved for `stuck_pattern`, `permission_failure`,
`tool_error_sequence` runs of 4+ errors, and `error_pattern` matches on
permission-denial keywords. Critical findings are sorted to the top of the
Recommendations table and rendered red.

**Related:** [`warning`](#warning), [`info`](#info).

### `warning`

**Short:** A finding worth investigating but not necessarily broken.

**Detail:** The default severity for outliers, retries, and short error
sequences. Warning findings often resolve on their own (e.g., a retry that
eventually succeeded) but warrant attention if they recur. Rendered yellow.

**Related:** [`critical`](#critical), [`info`](#info).

### `info`

**Short:** A passive observation ‚Äî usually a config-quality note.

**Detail:** Used for `mcp_unused_server` and most config-rubric
recommendations. These don't indicate runtime problems; they suggest
hygiene improvements. Rendered cyan.

**Related:** [`warning`](#warning), [`critical`](#critical).

## Confidence tier

Used by **delegation suggestions** (proposed new subagents derived from
recurring `general-purpose` invocations). Confidence reflects how reliably
the cluster represents a real, distinct task family vs. noise.

### `high`

**Short:** Cluster is large or cohesive enough to confidently propose as
a new subagent.

**Detail:** Triggered when **cluster_size ‚â• 10 OR cohesion_score ‚â• 0.50**.
The YAML draft is rendered without a "review before use" caveat ‚Äî the
cluster has enough internal similarity to read as a coherent task family.

**Related:** [`cohesion_score`](#cohesion_score), [`cluster_size`](#cluster_size), [`medium`](#medium), [`low`](#low).

### `medium`

**Short:** Cluster is real but borderline ‚Äî review before promoting.

**Detail:** Triggered when **cohesion_score ‚â• 0.35** but neither high-tier
threshold is met. The YAML draft is offered as a starting point; the
suggested name and prompt should be reviewed before saving the agent.

**Related:** [`high`](#high), [`low`](#low).

### `low`

**Short:** Cluster is loose; the proposed subagent draft needs rework.

**Detail:** Triggered when neither size nor cohesion thresholds are met.
The YAML draft includes a `# REVIEW BEFORE USE ‚Äî low confidence cluster`
comment. Use the cluster as evidence that *some* coherent subset of work
exists, not as a finished proposal.

**Related:** [`high`](#high), [`medium`](#medium), [`cohesion_score`](#cohesion_score).

## Recommendation target

Every recommendation names a single configuration surface to change. The
target maps the signal to the section of the agent's setup that needs work.

### `prompt`

**Short:** Edit the agent's prompt body ‚Äî the markdown content below the
frontmatter in `.claude/agents/<name>.md`.

**Detail:** Used for signals that suggest the agent's *instructions* are
the problem: token outliers (broad exploration), retry loops (missing
recovery guidance), error sequences (missing fallback). For built-in
agents (which have no editable prompt), the recommendation pivots to
either a wrapper subagent or a tighter delegation prompt ‚Äî see
[**Built-in agent concern: scope**](#scope) and [**recovery**](#recovery).

**Related:** [`tools`](#tools), [`model`](#model), [`mcp`](#mcp).

### `tools`

**Short:** Edit the agent's `tools:` list in frontmatter (or the
`disallowedTools:` list).

**Detail:** Used for `permission_failure` (the agent needs access to a
tool it doesn't have) and for token outliers when the agent has too many
tools to choose from (>8 ‚Üí recommendation forks to `tools`). For built-in
agents, see [**Built-in agent concern: tools**](#tools-1).

**Related:** [`prompt`](#prompt), [`permission_failure`](#permission_failure).

### `model`

**Short:** Change the agent's `model:` field in frontmatter.

**Detail:** Used for `model_mismatch` signals. Recommends `claude-haiku-4-5`
when overspec'd, `claude-sonnet-4-6` when underspec'd. For built-in agents
(model not configurable), pivots to "create a custom subagent with the
recommended model" ‚Äî see [**Built-in agent concern: model**](#model-1).

**Related:** [`model_mismatch`](#model_mismatch), [`complexity_tier`](#complexity_tier), [`overspec`](#overspec), [`underspec`](#underspec).

### `mcp`

**Short:** Edit MCP server configuration in `~/.claude.json` or `.mcp.json`.

**Detail:** Used for `mcp_unused_server` (remove or disable) and
`mcp_missing_server` (add the configuration). MCP recommendations carry
`agent_type: null` since MCP servers aren't scoped to a single agent.

**Related:** [`mcp_unused_server`](#mcp_unused_server), [`mcp_missing_server`](#mcp_missing_server).

## Built-in agent concern

Built-in Claude Code agents (Explore, Plan, general-purpose, code-reviewer,
statusline-setup, claude-code-guide) have no user-editable prompt, tool
list, or model selector. When a signal fires on a built-in agent, the
recommendation can't say "edit your prompt" ‚Äî there is no prompt to edit.
The **concern** field tells the user *what* to change instead.

### `scope`

**Short:** The agent's task scope is too broad.

**Detail:** Used for `token_outlier` and `duration_outlier` on built-in
agents. The recommendation suggests: (a) a custom wrapper subagent that
narrows the task, (b) a tighter delegation prompt passed to the built-in,
or (c) rerouting to a different agent.

**Related:** [`recovery`](#recovery), [`tools`](#tools-1), [`model`](#model-1).

### `recovery`

**Short:** The agent lacks error-recovery instructions.

**Detail:** Used for `retry_loop`, `tool_error_sequence`, `error_pattern`
on built-in agents. The recommendation suggests adding retry bounds or
exit conditions to the *delegating* agent's prompt (since the built-in
itself isn't editable), or wrapping the call in a custom subagent that
owns recovery logic.

**Related:** [`scope`](#scope), [`tools`](#tools-1).

### `tools`

**Short:** The agent's fixed tool set is wrong for the task.

**Detail:** Used for `permission_failure` on built-in agents. Suggests
routing the task to a custom subagent with explicit tool grants, or
confirming the built-in's fixed tool set is actually required.

**Related:** [`permission_failure`](#permission_failure), [`recommendation target: tools`](#tools).

### `model`

**Short:** The agent's fixed model is the wrong tier.

**Detail:** Used for `model_mismatch` on built-in agents. Suggests
creating a custom subagent with the recommended model or rerouting to an
existing custom agent that uses it.

**Related:** [`model_mismatch`](#model_mismatch), [`recommendation target: model`](#model).

## Cluster metrics

Used by the **delegation clustering** pipeline (`agentfluent[clustering]`
extra) when proposing new subagents from recurring `general-purpose`
invocations.

### `cohesion_score`

**Short:** Mean pairwise cosine similarity within a cluster (0.0‚Äì1.0).

**Detail:** Higher is more coherent. Real-world delegation prompts have
enough free-text variance that cohesion rarely exceeds 0.5 even for
recognizable task families. Empirically calibrated thresholds (#167):
**‚â• 0.50** = high confidence, **‚â• 0.35** = medium, below = low. The
metric is mean-pairwise rather than centroid-distance because clusters
of 5‚Äì10 invocations are sensitive to centroid placement.

**Example:** `cohesion_score: 0.42` on a five-invocation cluster about
"PR review prompts" ‚Äî solidly medium-confidence.

**Related:** [`cluster_size`](#cluster_size), [`confidence tier`](#confidence-tier).

### `cluster_size`

**Short:** Number of invocations grouped into a cluster.

**Detail:** Larger clusters are more reliable signals of a recurring
task family. **‚â• 10 invocations** is the high-confidence threshold;
below `--min-cluster-size` (default **5**) the cluster is suppressed
entirely (not enough evidence).

**Example:** `cluster_size: 12` ‚Äî confident enough to propose a new
subagent based on size alone.

**Related:** [`cohesion_score`](#cohesion_score), [`confidence tier`](#confidence-tier).

## Model routing

Used by the `model_mismatch` signal to characterize agents whose declared
model is wrong for their workload.

### `complexity_tier`

**Short:** A 3-level classification of an agent's task complexity.

**Detail:** Computed from observed mean tool calls, mean tokens, error
rate, and write-tool usage. Tiers map 1:1 to Claude model families:

- **`simple`** ‚Äî `claude-haiku` family. Triggered when mean tool calls
  ‚â§ 5 AND mean tokens ‚â§ 2,000 AND no write tools.
- **`moderate`** ‚Äî `claude-sonnet` family. Default tier when neither
  simple nor complex thresholds are met.
- **`complex`** ‚Äî `claude-opus` family. Triggered when mean tool calls
  ‚â• 10 OR mean tokens ‚â• 5,000 OR error rate ‚â• 0.20 OR write tools used.

**Calibration note (#140):** these thresholds are pending multi-contributor
validation. On v0.3 single-dataset calibration, every observed agent
classified as "complex" ‚Äî the thresholds may be set too sensitive.

**Related:** [`overspec`](#overspec), [`underspec`](#underspec), [`model_mismatch`](#model_mismatch).

### `overspec`

**Short:** The agent uses a more capable (more expensive) model than its
complexity warrants.

**Detail:** A `simple`-tier agent declared on `claude-opus-4-7`, for
example. The `model_mismatch` signal recommends switching to the
classified tier's family with an estimated cost savings derived from the
observed token volume.

**Related:** [`underspec`](#underspec), [`complexity_tier`](#complexity_tier).

### `underspec`

**Short:** The agent uses a less capable model than its complexity warrants.

**Detail:** A `complex`-tier agent declared on `claude-haiku-4-5`, for
example. May correlate with elevated `error_rate` or `retry_loop` signals
on the same agent. No cost-savings estimate (upgrading to a higher tier
costs *more*); the recommendation is purely about output quality.

**Related:** [`overspec`](#overspec), [`complexity_tier`](#complexity_tier).

## Built-in agent types

The closed set of agent types Claude Code provides out of the box. AgentFluent
recognizes these via case-insensitive match on `BUILTIN_AGENT_TYPES`. When
a signal fires on a built-in, recommendations use **agent concern**
templates (see above) rather than "edit your prompt" templates.

### `general-purpose`

The default delegation agent. Most recurring `general-purpose` invocations
in your sessions are clustering candidates ‚Äî these are tasks that probably
deserve their own custom subagent. See **delegation clustering** in the
README.

### `Explore`

Codebase exploration. Read-only by design. Frequent token outliers here
usually mean the delegation prompt is too open-ended.

### `Plan`

Planning agent ‚Äî produces step-by-step implementation plans without
modifying state.

### `code-reviewer`

PR / diff review agent.

### `statusline-setup`

Configures the Claude Code statusline.

### `claude-code-guide`

Documentation lookup for Claude Code itself (hooks, settings, MCP, etc.).

## Built-in tools

The most common tools observed in real Claude Code sessions. Not a closed
enum ‚Äî Claude Code can introduce new tools, and MCP servers contribute
additional namespaced tools (`mcp__<server>__<tool>`).

| Tool | Purpose |
|---|---|
| `Read` | Read a file by absolute path. |
| `Edit` | Targeted string-replacement edit on a file. |
| `Write` | Create or overwrite a file. |
| `Bash` | Run a shell command. State-changing; classified as a write tool. |
| `Grep` | Ripgrep-backed content search. Read-only. |
| `Glob` | File pattern match. Read-only. |
| `Agent` | Delegate a subtask to a subagent. The `subagent_type` input names which agent. |
| `TaskCreate` / `TaskUpdate` / etc. | Async task management. |
| `NotebookEdit` | Jupyter notebook cell edits. State-changing. |
| `WebFetch` | Fetch a URL. |
| `WebSearch` | Web search. |
| `Skill` | Invoke a slash-command-style skill. |
| `ExitPlanMode` | Transition out of plan mode. |
| `AskUserQuestion` | Pause and ask the user a structured question. |
| `ScheduleWakeup` | (Loop mode) schedule the next wake-up time. |

MCP tools follow `mcp__<server>__<tool>` (e.g., `mcp__github__create_issue`).
The server name segment can contain underscores; tool names can start with
underscore. The first `__` after the `mcp__` prefix is the server/tool
boundary.
